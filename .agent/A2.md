### 3. Can Android Home Screen Widgets be implemented without ejecting?
**Yes**, they can be implemented **without ejecting** (i.e., you do not need to migrate to a Bare React Native / React Native CLI setup). 

However, they **cannot run in the standard Expo Go client**. Because widgets require native code registrations (specifically Android `AppWidgetProvider`, native layout resource files, and background services), you must transition from running the app in Expo Go to running a custom **Development Build** (using Expo Prebuild to generate the native folders behind the scenes).

---

### 4. What is the recommended migration path?
The migration path is to move from **Expo Go** to **Expo Development Builds**:
1.  **Install the library**: Add a library that supports Expo Config Plugins (e.g., `react-native-android-widget`).
2.  **Add the config plugin**: Configure the library's plugin in [app.json](file:///c:/Users/admin/Documents/FPT_Source/SM26/MMA301/project/soccer-star-app/app.json) under `"plugins"`.
3.  **Implement JS widget handlers**: Define your widget's layouts and background task handlers in JavaScript/TypeScript. *Make sure to wrap these imports in platform checks (`Platform.OS === 'android'`) or dynamic `require()` statements so they do not crash on iOS or inside Expo Go.*
4.  **Prebuild & Run**: Run `npx expo prebuild` to generate the native folder structure locally, and use `npx expo run:android` to compile and run your custom development build on an emulator or physical device.

---

### 5. Is expo prebuild sufficient, or is a full React Native CLI migration required?
**Expo Prebuild is fully sufficient.** A full React Native CLI migration is **not** required. 

The config plugin will dynamically modify `AndroidManifest.xml`, inject widget layout configuration resources, and register the headless background service during the `prebuild` phase. This allows you to keep the project configuration managed entirely inside [app.json](file:///c:/Users/admin/Documents/FPT_Source/SM26/MMA301/project/soccer-star-app/app.json) and avoid committing `android/` or `ios/` folders.

---

### 6. Recommend the best library for implementing Android App Widgets in this project.
The recommended community-standard library is **`react-native-android-widget`**.
*   **Why**: It has a dedicated Expo Config Plugin (`react-native-android-widget/app.plugin`), a widget task handler for headless background updates, and React-like layout primitives (such as `FlexWidget`, `TextWidget`, and `ImageWidget`) designed specifically to map to Android's native `RemoteViews` layout system.

---

### 7. Explain the impact on OTA updates, EAS Build, and project maintenance.

*   **EAS Build**:
    *   **Fully Supported**: EAS Build executes `expo prebuild` automatically on the cloud build servers. The widget config plugin will run during the build and successfully compile the widget files into the final Android application package (`.apk` or `.aab`).
*   **OTA Updates (EAS Update)**:
    *   **Native Config Limits**: Changes to the native configuration of the widget (such as adding/removing a widget, changing its dimensions in `app.json`, or upgrading the native dependency version) **cannot** be updated via OTA updates. These require a new app store submission.
    *   **JS Logic Updates**: Because the widgets are updated using a background JavaScript task handler, updates to the business logic, formatting, and calculation of the widget's contents **can** be deployed via OTA (`eas update`), provided the underlying native layouts and configuration remain unchanged.
*   **Project Maintenance**:
    *   **Code Cleanliness**: Because Expo Prebuild handles the generation of native code, you don't need to commit the `android/` directory to Git. This prevents native code drift.
    *   **SDK Upgrades**: Future upgrades to newer Expo SDK versions will remain straightforward because you can simply run `npx expo install --fix` and let Expo re-generate the native folders cleanly.
    *   **Headless Context**: The widget updates run in a separate headless JS task. You need to ensure the JS logic is lightweight and does not assume a fully loaded React context (like access to React context providers, standard React hooks, or active navigation states).
